# 记忆工程宪法

## 1. 文档定位

本文档是记忆工程项目的最高级别说明文件，用来回答四个问题：

1. 这个项目到底要解决什么问题
2. 当前阶段真正优先做什么，不做什么
3. 所有策略和工程改动应该服从哪些核心目标
4. 当不同文档、不同模块、不同实现发生冲突时，应以什么原则裁决

本文档描述的是“项目的长期方向、当前阶段边界、核心判断原则”，不是具体实现手册，不替代数据结构文档、Prompt 文档或调试文档。

---

## 2. 项目名称

记忆工程 v2.0

---

## 3. 项目背景

记忆工程要构建的，不是一个简单的“照片分类器”，也不是一个只会生成标签的多模态流水线，而是一套面向个人长期记忆理解的基础系统。

系统输入是用户历史照片，输出不是单张图片的描述，而是围绕用户本人逐步建立出来的三类长期资产：

- 事件：用户经历过什么事
- 关系：用户和哪些人构成什么关系
- 画像：用户长期稳定的身份、生活方式、关系结构与表达特征

这些资产最终服务于一个更高层目标：

当系统未来再次看到用户的新照片时，能够基于过往记忆讲出“这张照片背后的故事”。
这个故事不仅要看懂画面，还要理解人物、关系、时间线、生活状态和社交语境，并最终让用户觉得“被理解”，让朋友觉得“有趣且愿意互动”。

---

## 4. 核心目标

本项目的核心目标不是“多做推断”，而是“建立可信、可复盘、可扩展的用户记忆底座”。

围绕这个目标，当前系统必须逐步获得四种能力：

- 可标注：用户未来可以显式修正人物、关系、画像字段
- 可追溯：每条结论可以回到照片、事件、关系和证据
- 可回写：修正后的结果可以覆盖旧结论并沉淀为系统事实
- 可约束：下游表达与推断不能绕过上游事实边界

这四个能力是项目北极星。

---

## 5. 当前阶段定位

当前阶段不是完整的记忆产品建设阶段，而是“主链路跑通 + 策略验证 + 观测性建设”阶段。

当前阶段最重要的事情有三件：

1. 跑通 `照片 -> 人脸 -> VLM -> 事件 -> 关系 -> 画像 -> 输出` 的主链路
2. 保证每一步中间结果足够可观察、可检查、可复盘
3. 快速验证哪些策略有效，哪些策略会造成误判、漏判和链路断裂

这意味着，当前阶段首先服务的是：

- 产品经理调策略
- 工程侧查问题
- 团队做复盘
- 为下一阶段的标注、回写、追溯打基础

---

## 6. 当前阶段非目标

在当前阶段，下列事项都不是最高优先级：

- 完整的用户标注产品
- 完整的 memory store 基础设施
- 全量历史事实融合与覆盖系统
- 完整的跨模块追溯平台
- 为兼容旧版本而保留复杂的历史实现
- 为了“看起来聪明”而做不可验证的自由推断

如果某个方案会损害主链路清晰度、观测性、可复盘性，即使它理论上更完整，也不应优先落地。

---

## 7. 目标用户

系统当前主要面向美国高中生和大学生。

这是一个高社交密度、关系变化快、生活场景丰富、身份切换频繁的人群。其照片中常同时包含：

- 高频社交关系
- 快速变化的恋爱与搭子关系
- 校园、活动、聚会、旅行、消费、表达等混合场景
- 大量“同框不等于关系稳定”的噪声人物

因此，本项目不能把“看见人物”简单等价为“认识关系”，也不能把“高光事件”简单等价为“长期事实”。

---

## 8. 系统核心产物

记忆工程当前沉淀三层核心产物：

### 8.1 事件

事件用于回答：用户最近和长期都经历了什么。
事件是后续关系识别和画像生成的重要上游。

### 8.2 关系

关系用于回答：用户与谁构成何种社会关系。
关系不是装饰层，而是画像层的重要约束条件。

### 8.3 画像

画像用于回答：用户是一个怎样的人，处在什么生活状态，具有什么长期稳定特征与短期变化特征。
画像必须建立在事件和关系之上，不能脱离上游事实自由漂移。

---

## 9. 核心方法论

### 9.1 代码做重活，LLM 只做判定

所有可以通过确定性逻辑完成的事情，优先交给代码，不交给 LLM。
例如：频率统计、时间跨度、异常检测、亲密度基线、候选过滤。

### 9.2 证据闭环

所有正式结论都必须有可观测证据支撑。
没有足够证据时，宁可输出 `null`、降级、或标记为待观察，也不允许无根据补全。

### 9.3 用户本人是全链路圆心

主角识别、关系识别、画像生成，最终都必须围绕“用户本人”组织。
任何脱离主角语境的推断，都不是正式记忆结论。

### 9.4 上游约束下游

事件是关系的基础，关系是画像的重要边界。
画像层不能绕过关系层直接伪造用户长期社会事实。

### 9.5 当前优先保证可观察，而不是一次性做全

当前阶段宁可保留更多中间产物，也不优先追求“端到端黑盒一把出答案”。

---

## 10. 治理闭环与职责边界

### 10.1 系统已经是“双环结构”

记忆工程当前不再只是单一生成链路，而是一个双环系统：

- 生产环：`照片 -> 人脸 -> VLM -> 事件 -> 关系 -> 画像 -> 输出`
- 治理环：`badcase -> case -> pattern -> task -> proposal -> engineering change -> outcome`

生产环负责生成正式记忆结果。
治理环负责发现问题、沉淀问题、审批问题、修正规则，并把修正结果安全落回工程。

任何对项目的管理、评估和资源分配，都必须同时考虑这两条链，而不能只看前者。

### 10.2 系统可以自动做的事

在当前阶段，系统可以自动完成以下事项：

- 自动运行主链路，生成事件、关系、画像和正式输出
- 自动运行下游审计，并在单次运行内执行主角、关系、画像三类 backflow
- 自动在 backflow 后触发必要 rerun，得到本轮最终正式结果
- 自动把 mainline 与 downstream audit 的问题沉淀成 case
- 自动做 triage、pattern clustering、difficult case 识别和 engineering alert 识别
- 自动生成 strategy experiment、proposal 和 engineering change request 等候选材料
- 在“工程执行审批已通过”的前提下，自动把 patch 写入 repo 规则资产，并执行最小验证

### 10.3 必须由人类管理者批准的事

以下事项不能由系统自己做最终决策，必须由产品、策略或工程负责人批准：

- 选择问题应该修哪一面，例如判断口径、找证据规则、调用时机，还是转工程问题
- 确认某次实验结果是否有资格进入正式规则
- 确认某次正式规则改动是否真正执行到仓库
- 确认 engineering issue 是否进入工程修复通道
- 确认 difficult case 是继续观察、补证据，还是转人工专项处理
- 对 reject / need_revision 类审批给出明确 reviewer note

### 10.4 系统明确不能做的事

在当前阶段，系统不得自动执行以下事项：

- 自动批准自己的 proposal
- 自动批准 engineering execute
- 绕过审批，直接把反思结果发布成正式 repo 规则
- 自动修改任意 Python 业务逻辑
- 自动执行多字段、多改面的正式规则发布
- 把 GT 直接当作在线主链路事实覆盖正式输出
- 把 difficult case 强行自动归因成正式改动结论
- 在通知未发送或审批未发生时，把任务视为已经完成审批

### 10.5 当前阶段的人类管理原则

人类管理者在这套系统中的职责，不是逐条替系统做判断，而是把住四道关：

- 目标关：判断这条问题值不值得修
- 方向关：判断应该修哪一面，而不是让系统自己决定发布面
- 发布关：判断候选实验是否能进入正式规则
- 风险关：判断当前改动是否超出“最小改面、单字段、可回滚、可验证”的边界

系统拥有运行期纠偏权，但不拥有规则发布权。
系统拥有问题发现权，但不拥有人类决策替代权。

---

## 11. 成功标准

如果这个项目当前阶段做对了，应该满足以下判断：

- 我们能清楚知道每一步输入输出是什么
- 我们能定位关系误判来自哪里
- 我们能定位画像漂移来自哪里
- 我们能清楚区分“真实问题”和“观测缺失”
- 我们能在不推翻整个系统的前提下，逐步替换局部策略
- 我们能把 badcase 稳定沉淀为 case，而不是停留在一次性人工讨论
- 我们能让提案、审批、执行和 outcome 形成可复盘闭环
- 我们能确保跨运行的正式规则变更必须经过人工批准，而不是由系统自行发布

---

## 12. 本文档的约束级别

本文档是项目最高层级原则文件。

当以下内容发生冲突时，优先级应为：

1. 宪法
2. 当前主链路真实数据流与正式输出约束
3. 阶段性方案文档
4. 实现细节
5. 历史文档

任何新方案如果与本文档冲突，默认视为需要重新审查，而不是直接合入。
