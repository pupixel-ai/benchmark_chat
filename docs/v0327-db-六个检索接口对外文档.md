# v0327-db-query 六个检索接口对外文档

默认对外版本：`v0327-db-query`  
文档语言：中文  
面向对象：下游系统、接口调用方、数据消费方

> 本文档覆盖 6 个正式对外检索接口：
> `/memory/faces`、`/memory/events`、`/memory/vlm`、`/memory/profiles`、`/memory/relationships`、`/memory/bundle`。
> 自本版起，接口统一改为 `task-scoped` 设计：`task_id` 进入 URL path，不再使用 `user_id` 路由。
> `memory/core` 不再作为对外文档主口径。
> 当前生产服务同时支持 `v0327-db` 与 `v0327-db-query` 任务，但下游默认应按 `v0327-db-query` 版本接入。
> 如果下游需要新建任务或默认配置任务版本，请直接写 `v0327-db-query`。

## 1. 统一约定

### 1.1 默认服务地址

当前默认服务部署在 `app-v0317`，内网 IP 为 `10.60.1.243`。

- 前端默认地址：`http://10.60.1.243:3000`
- 后端默认地址：`http://10.60.1.243:8000`
- 本文档中的接口调用示例，如需写成完整 URL，默认都基于上述后端地址。

### 1.2 路由前缀

所有接口都挂在：

```http
GET /api/tasks/{task_id}/memory/*
```

其中：

- `task_id`：唯一任务标识，直接决定本次检索范围
- 登录账号：仅用于会话鉴权，不是被检测的人脸用户 ID
- 被检测的人脸/照片所属用户：属于任务内容，不通过 URL 传入

### 1.3 认证方式

所有接口都要求先登录，再携带会话 Cookie 访问。

推荐联调顺序：

1. `POST /api/auth/login`
2. `GET /api/auth/me`
3. `GET /api/tasks`
4. 新建任务时默认使用 `v0327-db-query`
5. 选择目标 `task_id`
6. 调用本文档中的 6 个检索接口

### 1.3.1 响应压缩

当前生产服务已开启 `gzip` 响应压缩。

- 客户端默认无需额外改造
- 常见客户端如 `requests`、`curl --compressed`、浏览器都会自动带 `Accept-Encoding: gzip`
- 对于 `events / relationships / bundle` 这类大 JSON 接口，服务端会在响应体超过阈值时自动压缩
- 下游如果需要排查网络问题，请优先检查响应头里的 `Content-Encoding: gzip`

### 1.4 通用查询参数

由于接口改为 `task-scoped`，不再支持通过 `user_id + dataset_id + all + scope` 组合选任务。  
当前每次请求只查询 URL 指定的单个任务，因此下面只保留对返回内容有影响的可选参数：

| 参数 | 说明 |
|---|---|
| `include_raw` | 是否返回原始阶段数据。默认 `false`。 |
| `include_artifacts` | 是否返回 artifacts。默认 `false`。 |
| `include_traces` | 是否返回 agent traces。默认 `false`。 |

### 1.5 统一响应壳

```json
{
  "query": {
    "task_id": "task_xxx",
    "include_raw": false,
    "include_artifacts": false,
    "include_traces": false
  },
  "task": {},
  "data": {},
  "artifacts": null
}
```

说明：

- `query.task_id`：当前命中的任务
- `task`：任务级元信息
- `data`：当前接口的主返回体
- `artifacts`：仅在 `include_artifacts=true` 时返回

### 1.6 照片与人脸 URL 约定

- 凡是返回照片对象，都应返回稳定可展示 URL
- 凡是返回人脸对象，都应返回裁图 URL
- 下游**优先直接消费接口返回的 URL 字段**
- 下游**不要根据 `photo_id` 或 `face_id` 自己猜路径**
- 当前生产环境中，照片 URL 的真实格式以接口返回值为准，常见形态例如：

```text
/api/assets/5570845cce1a47819227b3d89fcec9cb/uploads/001_MVIMG_20260118_135245.jpg
```

推荐消费顺序：

1. 如果响应中有 `photo_url`，直接用 `photo_url`
2. 如果响应中有 `urls.display`，优先用 `urls.display`
3. 如果响应中有 `cover_photo_url` 或 `photo_urls`，直接使用这些字段
4. 如果 `urls.crop` 为空，使用 `urls.boxed_full` 或对应照片的 `photo_url`
5. 不要自己拼接 `/api/assets/...`

### 1.7 六个正式接口

```http
GET /api/tasks/{task_id}/memory/faces
GET /api/tasks/{task_id}/memory/events
GET /api/tasks/{task_id}/memory/vlm
GET /api/tasks/{task_id}/memory/profiles
GET /api/tasks/{task_id}/memory/relationships
GET /api/tasks/{task_id}/memory/bundle
```

### 1.8 当前线上验证结果（2026-03-30）

以下结论基于 `app-v0317` 当前线上服务 `http://10.60.1.243:8000` 的实测结果：

| 任务版本 | task_id | faces | events | vlm | profiles | relationships | bundle | 说明 |
|---|---|---:|---:|---:|---:|---:|---:|---|
| `v0327-db` | `5570845cce1a47819227b3d89fcec9cb` | 200 | 200 | 200 | 200 | 200 | 200 | 全通，返回完整数据 |
| `v0327-db-query` | `e79a9632744a4a7f801bacae7fddf012` | 200 | 200 | 200 | 200 | 200 | 200 | 路由全通，但该历史任务只有 face 数据，后 5 个接口返回空数组 |

补充说明：

- 以上 6 个接口现在都已经部署到生产环境，不再存在“文档里有、线上没路由”的情况。
- `v0327-db-query` 这条历史任务 `e79...` 不是服务坏了，而是任务本身没有跑出 VLM / 事件 / 关系 / 画像数据，所以后 5 个接口返回的是空结果。
- `faces` 的首张 `photo_url` 与 `v0327-db` / `v0327-db-query` 两条任务都已实测可直接取图，返回 `200 image/jpeg`。
- 对外默认版本仍然是 `v0327-db-query`；表格里保留 `v0327-db` 只是因为当前这条历史任务 `557...` 具备完整样例数据，适合做只读联调示例。

### 1.9 当前数据源说明

当前 6 个接口已经统一为“读表优先”的方式：

- `faces`
  - 读取 `memory_photos`、`memory_persons`、`memory_faces`
  - 这些表里的数据来源于 face 阶段输出 `face_recognition`
  - 也就是说，`face_recognition` 仍然是人脸阶段原始快照，但线上 serving 已经不再直接拆原始 JSON，而是先物化进表再读
- `events`
  - 读取 `memory_events`、`memory_event_photos`、`memory_event_people`、`memory_event_places`
  - 响应只返回 `events`
  - 每个 `event` 自己携带 `photos / photo_ids / photo_urls / cover_photo_url`
- `vlm`
  - 读取 `memory_evidence`
- `profiles`
  - 读取 `memory_profile_facts`
- `relationships`
  - 读取 `memory_relationships`、`memory_relationship_support`
- `bundle`
  - 读取上述表并拼装完整返回

对下游来说，可以直接理解成一句话：

> 这 6 个接口现在读取的是数据库里的结构化表，不是直接拆 `task.result.memory`。

但有一个前提：

> 如果某条任务历史上根本没跑出某阶段数据，那么对应表里也不会有内容，所以接口会返回空数组，而不是凭空补造数据。

## 3. 事件接口

**接口**：`GET /api/tasks/{task_id}/memory/events`  
**用途**：返回该任务的 LP1 正式事件结果，可选附带 `raw_events`。

**典型请求**：

```http
GET /api/tasks/5570845cce1a47819227b3d89fcec9cb/memory/events?include_raw=true
```

**当前正式口径**：

- 顶层只返回 `data.events`
- 不再顶层返回整条任务的全量 `photos / persons`
- 每个 `event` 自带自己的 `photos / photo_ids / photo_urls / cover_photo_url`
- `include_raw=true` 时返回 `data.raw_events`

**详细文档**：

- 详版 `Events API` 文档见：
  [`V0327_DB_QUERY_EVENTS_API_CN.md`](/Users/ziyan/Downloads/memory_engineering_v1.0.1_20260310_v0327_db_query/docs/V0327_DB_QUERY_EVENTS_API_CN.md)

**当前线上实测**：

- 样例任务：`5570845cce1a47819227b3d89fcec9cb`
- 非本机调用耗时：约 `5.533s`
- 响应体大小：约 `902KB`
- `event_count = 152`
- 顶层 `data` 只有 `events`

## 4. VLM 接口

**接口**：`GET /api/tasks/{task_id}/memory/vlm`  
**用途**：返回该任务的 VLM 级照片理解结果，用于消费照片观察、OCR、场景、互动等结构化字段。

**典型请求**：

```http
GET /api/tasks/5570845cce1a47819227b3d89fcec9cb/memory/vlm
```

**返回重点**：

- `data.photos[]`
- `data.vlm_items[]`
- 每张照片关联稳定 `photo_url`
- 可选返回 OCR、scene、people、interaction 等原始字段

## 8. 设计结论

本版对外契约统一采用 `task-scoped`：

- 路由唯一主键是 `task_id`
- 不再通过 `user_id` 路由任务
- `user_id` 只存在于服务端登录态与权限校验中
- 如果下游需要跨任务汇总，应先调用 `GET /api/tasks` 选出目标任务，再逐个请求本页 6 个接口

这套设计的直接好处是：

- 路由与数据边界一致
- 权限模型更清晰
- 不需要 `task_id + user_id + dataset_id + all + scope` 共同决定命中任务
- 下游调试和排障更简单

如果只是先做一轮稳定联调，推荐优先使用：
