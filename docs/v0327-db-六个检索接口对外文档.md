## 1. 核心口径

### 1.1 身份语义

从本版本开始，接口与数据库统一采用下面这套语义：

- `user_id`
  - 指照片主人 / 被分析对象 / subject user
  - 下游拿数据时，以这个 `user_id` 为主键
- `operator_user_id`
  - 指代上传照片或代操作的后台管理员
  - 这是可选元信息，不是接口主键
- self-upload 场景
  - 用户自己上传时，只有 `user_id`
  - 没有 `operator_user_id`

一句话说：

> 用户账户、后台管理员账户、照片主人，这三个概念不再混用。

### 1.2 默认版本

对下游的默认接入版本是：

```text
v0327-db-query
```

如果下游需要新建任务、写默认配置或写联调脚本，请始终默认使用 `v0327-db-query`。

### 1.3 历史保留样例

历史任务 `5570845cce1a47819227b3d89fcec9cb` 保留原有 `task_id / user_id`，作为只读联调样例使用。

这是一条明确的历史例外：

- 它可以继续用于只读联调和回归验证
- 但不代表新任务继续沿用旧的错误语义

## 4. 正式路由

### 4.1 任务列表与详情

```http
GET /api/users/{user_id}/tasks
GET /api/users/{user_id}/tasks/{task_id}
```

说明：

- path 里的 `user_id` 是照片主人
- 不是当前登录管理员
- 同一个管理员可以代多个不同 `user_id` 上传任务
- 下游应始终以 `user_id -> tasks -> task_id` 这条链组织数据

### 4.2 六个正式检索接口

```http
GET /api/users/{user_id}/tasks/{task_id}/memory/faces
GET /api/users/{user_id}/tasks/{task_id}/memory/events
GET /api/users/{user_id}/tasks/{task_id}/memory/vlm
GET /api/users/{user_id}/tasks/{task_id}/memory/profiles
GET /api/users/{user_id}/tasks/{task_id}/memory/relationships
GET /api/users/{user_id}/tasks/{task_id}/memory/bundle
```

### 4.3 Query API

```http
POST /api/users/{user_id}/tasks/{task_id}/memory/query
```

## 5. 创建任务的正式语义

当前创建任务仍走：

```http
POST /api/tasks
```

但 `request body` 的语义已经改成：

- `user_id`
  - 指照片主人 / subject user
  - 管理后台代上传时必须传
- 当前登录账号
  - 作为 operator 使用
  - 服务端自动记录为 `operator_user_id`
- 如果不传 `user_id`
  - 默认视为 self-upload
  - 即当前登录用户既是操作者，也是照片主人

管理员代上传示例：

```json
{
  "version": "v0327-db-query",
  "user_id": "jennie_user_id",
  "normalize_live_photos": true
}
```

## 6. 通用响应壳

```json
{
  "query": {},
  "task": {},
  "data": {},
  "artifacts": null
}
```

其中：

- `query`
  - 当前请求的 path/query 参数回显
- `task`
  - 当前任务元信息
- `data`
  - 当前接口主数据
- `artifacts`
  - 仅在 `include_artifacts=true` 时返回

## 7. 通用查询参数

| 参数 | 说明 |
| --- | --- |
| `include_raw` | 是否返回原始阶段数据，默认 `false` |
| `include_artifacts` | 是否返回 artifacts，默认 `false` |
| `include_traces` | 是否返回 traces，默认 `false` |

## 8. 照片 URL 约定

下游**优先直接消费 API 返回的 URL**，不要自己根据 `photo_id` 猜路径。

推荐使用顺序：

1. `photo_url`
2. `urls.display`
3. `cover_photo_url`
4. `photo_urls`

## 9. 响应压缩

生产服务已开启 `gzip` 响应压缩。

- 客户端一般无需额外改造
- `events / relationships / bundle` 这类较大 JSON 会自动压缩
- 如需排查网络问题，请记录：
  - `Content-Encoding`
  - `Content-Length`
  - 总耗时

## 10. 六个接口的职责边界

### 10.1 faces

```http
GET /api/users/{user_id}/tasks/{task_id}/memory/faces
```

职责：

- 返回 face 阶段的人脸结果
- 返回 `photos / persons / faces`
- 每张照片、人脸都带稳定 URL

### 10.2 events

```http
GET /api/users/{user_id}/tasks/{task_id}/memory/events
```

职责：

- 返回该任务的事件列表
- 顶层只返回 `data.events`
- 每个 event 自带自己的 `photos / photo_ids / photo_urls / cover_photo_url`
- 不再顶层返回整条任务的全量 `photos / persons / evidence`

详版见：
- [V0327_DB_QUERY_事件API.md](/Users/ziyan/Downloads/memory_engineering_v1.0.1_20260310_v0327_db_query/docs/V0327_DB_QUERY_事件API.md)

### 10.3 vlm

```http
GET /api/users/{user_id}/tasks/{task_id}/memory/vlm
```

职责：

- 返回 VLM 观察结果
- 用于消费 OCR、场景、互动、照片级理解

### 10.4 profiles

```http
GET /api/users/{user_id}/tasks/{task_id}/memory/profiles
```

职责：

- 返回画像结果
- 主要消费 `profile_facts / structured profile`

### 10.5 relationships

```http
GET /api/users/{user_id}/tasks/{task_id}/memory/relationships
```

职责：

- 返回人物关系结果
- 主要消费 `relationship_type / status / confidence / intimacy_score`

### 10.6 bundle

```http
GET /api/users/{user_id}/tasks/{task_id}/memory/bundle
```

职责：

- 返回 faces / events / vlm / profiles / relationships 的全量聚合包
- 适合离线消费或工作台一次拉全量

## 11. 当前数据来源

现在这 6 个接口都按“结构化表优先”读取：

- `faces`
  - `memory_photos`
  - `memory_persons`
  - `memory_faces`
- `events`
  - `memory_events`
  - `memory_event_photos`
  - `memory_event_people`
  - `memory_event_places`
- `vlm`
  - `memory_evidence`
- `profiles`
  - `memory_profile_facts`
- `relationships`
  - `memory_relationships`
  - `memory_relationship_support`
- `bundle`
  - 聚合上述表

这意味着：

- 不再依赖“大 JSON 读接口时临时拆”
- 每个阶段产出后即写表
- 接口直接读表返回

## 13. 推荐下游接入方式

### 13.1 正式主键

下游应以 `user_id` 为主键组织数据。

推荐流程：

1. 登录
2. `GET /api/users/{user_id}/tasks`
3. 选定 `task_id`
4. 调用六个检索接口或 Query API

### 13.2 管理后台代上传

如果是管理后台代上传：

- 登录态属于管理员
- `POST /api/tasks` 时传 `user_id = 照片主人`
- 后续查询一律走照片主人的 `user_id`
- 不要用管理员账号 ID 作为数据主键

### 13.3 self-upload

如果是用户自己上传：

- `POST /api/tasks` 可不传 `user_id`
- 服务端会自动把当前登录用户当作 subject user
- 此时没有 `operator_user_id`
